## Building & signing â€” READ THIS FIRST

**Always build with `./build.sh`.** It signs with the stable identity pinned in `.signing-identity`
(gitignored, machine-local) â€” now the **Developer ID Application** cert (team `ZWBJJX7NM9`). Do **NOT**
produce an ad-hoc build for the user to run.

Why it matters: the app needs the macOS **Accessibility (TCC)** permission. TCC keys the grant on the
code-signing *designated requirement*. For a Developer ID app the DR is keyed on the **Team ID**
(`leaf[subject.OU] = ZWBJJX7NM9`), not the specific cert â€” so dev rebuilds, releases, and even a re-issued
Developer ID cert all share **one** requirement â‡’ the grant **survives** and you don't get duplicate
Accessibility entries. Using a *different* identity for dev vs release (Apple Development locally, Developer
ID for releases) is exactly what once littered TCC with stale duplicate entries â€” keep both on the **same**
Developer ID. **Ad-hoc** signing (`SIGN_IDENTITY=-`, the bare-`./build.sh` default *without*
`.signing-identity`) changes the code hash every build â‡’ the grant is dropped each rebuild. Keep
`.signing-identity` in place; if it's missing, recreate it (`security find-identity -v -p codesigning`).

- Plain **`./build.sh`** = fast local build (sign only, no network, no notarization).
- **`RELEASE=1 ./build.sh`** = notarized + stapled app and a notarized **DMG** (`build/AltTab.dmg`) for a
  GitHub Release. Needs the `notarytool` profile `AltTabNotary` in the keychain; `DMG=0` skips the image.

## Runtime / deployment

- The user runs **`/Applications/AltTab.app`** (they move/replace `build/AltTab.app` there themselves).
  Because the DR is keyed on the Team ID (Developer ID), replacing the bundle with any freshly
  Developer-ID-signed build (dev or release) keeps the grant.
- It's an `LSUIElement` agent (menu-bar only, no Dock tile). Bundle id: `dev.fusy.alttab`.
- App icon: `Resources/AltTab.icns`, generated by `scripts/makeicon.swift` (CoreGraphics, no asset catalog).

## If the Accessibility grant "resets" / native Cmd+Tab comes back

Symptom: the default Apple Cmd+Tab appears = AltTab isn't disabling the native switcher = it lacks
Accessibility (the whole stack is gated behind it). Checklist:
1. Confirm the **running** bundle is the stably-signed one: `codesign -dv --verbose=4 <app>` (look for a
   real Authority, not "adhoc") and `codesign -d -r- <app>` (DR should reference the cert leaf, not a cdhash).
2. Clear stale/conflicting TCC entries, then re-grant **once**:
   `tccutil reset Accessibility dev.fusy.alttab`
3. Verify objectively: AltTab disables symbolic hotkeys 1 (Cmd+Tab) & 2 (Cmd+Shift+Tab) when active â€”
   probe via `CGSIsSymbolicHotKeyEnabled`/`SLSIsSymbolicHotKeyEnabled` (1 = native ON / AltTab not
   overriding; 0 = AltTab active).

## Debugging / logs

The app logs to **unified logging**, subsystem `dev.fusy.alttab` (`Log.swift`; categories: app,
hotkeys, session, store, focus). Default/error-level events persist on disk â€” diagnose AFTER a repro:
`./scripts/logs.sh 10m` dumps the last 10 min to a file; bare `./scripts/logs.sh` streams live
(incl. debug-level, which is memory-only and absent from dumps). Crash reports (none so far) would
land in `~/Library/Logs/DiagnosticReports/`. Key persisted events: per-summon window count
("summon bailed" = store got gutted), reconcile drops in WindowStore (only when the WindowServer no
longer lists the window â€” kAXWindows absence just means "other Space" and KEEPS it, debug-logged),
event-tap disabled-by-OS, Focus fallbacks.
